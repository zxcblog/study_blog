
# etcd写请求执行流程
```shell
etcdctl put test test --endpoints 127.0.0.1:2379
```

![etcd写流程](../img/etcd写流程.png)

1. client端通过负载均衡算法选择一个etcd节点，发起grpc调用
2. etcd节点收到请求后经过grpc拦截器、Quota模块后，进入KVServer模块
3. KVServer模块向Raft模块提交一个提案，提案内容为"大家好，请使用PUT方法执行一个key为test value为test的命令"
4. 随后此提案通过RaftHTTP网络模块转发、经过集群多数节点持久化后，状态会变成已提交
5. etcd server从Raft模块获取已提交的日志条目，传递给apply模块
6. apply模块通过mvcc模块执行提案内容，更新状态机。

与读流程不同的是， 写流程还涉及quota、wal、apply三个模块，etcd的crash-safe及幂等性也正是基于wal和apply流程的consistent index等实现的

## Quota模块
该模块主要用于检查当前etcd db 大小加上请求的key-value大小之和是否超过了配额（quota-backend-bytes）, 如果超过了配额，他会产生一个告警（alarm）请求，
告警类型是no space，并通过raft日志同步给其他节点，告知db没有空间了，并将告警持久化存储到db中，最终，无论是api层grpc模块还是负责将raft侧已提交的日志条目应用
到状态机的apply模块，都拒绝写入，集群只读。   
常见的etcd server：mvcc database space exceeded错误就是因为 quota模块检测到db大小超限导致的。
- 默认配额仅为2GB, 当业务数据，写入qps、kubernetes集群规模增大后，etcd 的大小可能会超过2GB
  - 调大配额， etcd社区建议不超过8GB， 如果填写的是个小于0的树，会禁用配额功能，这可能会让db大小处于失控，导致性能下降，不建议禁用配额
- etcd是mvcc数据库，保存了key的历史版本，当未配置压缩策略的时候，随着数据不断写入，db大小不断增大，导致超限
  - 压缩时只会给旧版本key打上空闲（free）标记，后续新的数据写入的时候可复用这块空间，db大小并不会减小
  - 如果需要回收空间，减少db大小，得使用碎片整理（defrag），它会遍历旧的db文件，写入到一个新的db文件。但是它对服务性能有较大的影响，不建议在生产集群频繁使用

调整后还需要手动发送一个取消告警（etcdctl alarm disarm）的命令， 以取消所有告警，否则因为告警的存在，集群还是无法写入

## KVServer模块
通过流程二的配额检查后，请求就从api层转发到流程三的KVServer模块的put方法。KVServer模块主要功能为
- 打包方案： 将put写请求内容打包成一个提案消息，提交给raft模块
- 请求限速、检查：不过在提交前还有限速、鉴权和大包检查

### prefilght check
为了保证集群稳定性、避免雪崩，任何提交到raft模块的请求，都会做一些简单的限速判断
- 限速：如果raft模块已提交的日志索引（committed index）比已应用到状态机的日志索引超过了5000， 会返回一个 `etcdserver: too many requests`错误给client
- 鉴权：尝试去获取请求中的鉴权消息，若使用了密码鉴权，请求中携带了token，如果token无效，会返回`auth:inivaild auth token`错误给client
- 大包检查： 检查写入的包大小是否超过默认的1.5MB， 如果超过了会返回`etcdserver:request is too large`的错误给client

### propose
通过检查后会生产一个唯一的ID，将此请求关联到一个对应的消息通知channel（用于接收结果），然后向raft模块发起一个提案
向raft模块发起提案后，kvserver模块会等待此put请求，等待写入结果通过消息通知channel返回或者超时。etcd默认超时时间为7秒，（5秒磁盘IO延时+2*1秒竞选超时时间），如果请求超时未返回结果，则可能会出现`etcdserver: request timed out`错误
